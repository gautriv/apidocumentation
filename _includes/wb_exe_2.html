Exercise 2: Documenting Multiple Webhook Events

Scenario

You are documenting a payment processing platform's webhooks that notify merchants about different payment events.

Your Task

Create comprehensive documentation for multiple payment-related webhook events:

  • Payment success
  • Payment failure
  • Refund processed
  • Subscription renewal

For each event, document:

  • Event name and trigger conditions
  • Payload structure with detailed field descriptions
  • Common use cases for the event

Webhook Event Structure

All webhook events follow this basic structure:

{
    "event_type": "[EVENT_TYPE]",
    "timestamp": "2023-07-20T15:31:49Z",
    "version": "1.0",
    "data": {
        // Event-specific data
    }
}

Example Event Types

  • payment.succeeded
  • payment.failed
  • refund.processed
  • subscription.renewed
Bonus Challenge

Add a section on "Migrating Between Webhook Versions" that explains how developers should handle changes to webhook payload structures over time.

Payment Processing Platform Webhooks

Our webhook system allows you to receive real-time notifications about payment events. By subscribing to these webhooks, you can automate your business processes like order fulfillment, inventory management, and customer notifications.

Common Webhook Properties

All webhook events share these common properties:

Property Type Description
event_type String Identifies the type of event that occurred
timestamp String ISO 8601 timestamp when the event occurred
version String API version of the webhook payload
data Object Event-specific data payload

1. Payment Success Event

Event Details

Event Type: payment.succeeded

Triggered When: A payment is successfully processed and funds are secured

Common Use Cases: Order fulfillment, inventory updates, sending receipts

Example Payload:
{
    "event_type": "payment.succeeded",
    "timestamp": "2023-07-20T15:31:49Z",
    "version": "1.0",
    "data": {
        "payment_id": "pay_8fh489f94hf984hf9h4",
        "amount": 42.99,
        "currency": "USD",
        "status": "succeeded",
        "payment_method": {
            "type": "card",
            "last4": "4242",
            "brand": "visa",
            "exp_month": 12,
            "exp_year": 2025
        },
        "customer": {
            "id": "cus_9f84h984hf984hf",
            "email": "customer@example.com"
        },
        "metadata": {
            "order_id": "order_98f4h89h4f948"
        }
    }
}
Property Type Description
payment_id String Unique identifier for the payment
amount Number Amount that was charged
currency String Three-letter ISO currency code
status String Current status of the payment
payment_method Object Details about the payment method used
customer Object Information about the customer
metadata Object Additional data attached to the payment

Triggered When: A payment attempt fails due to card decline, insufficient funds, or other reasons

Common Use Cases: Notifying customers of failed payments, retrying payments, order status updates

        "status": "failed",

            "decline_code": "insufficient_funds"

5. Migrating Between Webhook Versions

Version Migration Strategy

When we make changes to our webhook payload structure, we follow these principles:

  • We never remove fields from existing webhook payloads in the same version
  • New fields may be added to existing webhook versions in a backward-compatible way
  • Major structural changes are introduced in new API versions
Version Compatibility Example:

When processing webhooks, check the version field to determine the payload structure:

function processWebhook(payload) {
    const version = payload.version;
    
    if (version === "1.0") {
        // Handle version 1.0 payload structure
    } else if (version === "2.0") {
        // Handle version 2.0 payload structure
    } else {
        // Handle unknown version
        console.warn(`Unknown webhook version: ${version}`);
        // Use best-effort processing with defaults
    }
}
Webhook Version Timeline

Version 1.0: Current version

Version 2.0: Planned for Q1 2024 with enhanced metadata and customer objects

Each version will be supported for at least 12 months after a new version is released. We recommend building your integration to check the version field and handle multiple versions.
